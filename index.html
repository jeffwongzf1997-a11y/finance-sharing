基础概念与工具准备

30秒理解：SQL 和 Python 是什么？

SQL — 向数据库"提问"的语言

SQL就像是对数据库说的"普通话"。你用它告诉数据库："我想看什么数据"。

举个例子，你想知道"2024年HK市场每个活动发了多少钱"，用SQL就是： 

SELECT act_title, SUM(send_amt_hkd)
FROM 发奖流水表
WHERE year = '2024'
GROUP BY act_title

翻译成人话：从发奖流水表里，按活动名称汇总发奖金额，只看2024年的数据。

Python — 让电脑自动干活的语言

Python就像是给电脑写的"操作指令"。你用它告诉电脑："帮我自动做这些事"。

举个例子，你想把10个Excel文件合并成一个： 

import pandas as pd
import glob

files = glob.glob("*.xlsx")
result = pd.concat([pd.read_excel(f) for f in files])
result.to_excel("合并结果.xlsx")

翻译成人话：读取当前目录下所有xlsx文件，合并在一起，保存为"合并结果.xlsx"。

你不需要记住这些代码！ 以上代码只是让你对SQL和Python有个直观感受。实际使用中，你只需要用中文告诉AI你想做什么，AI会帮你写好代码。你要做的只是把AI写好的SQL复制到FDS平台运行，或者让AI直接运行Python脚本。

案例 5 编写复杂检测逻辑 — 10种重复发奖检测

工作场景：检测通用新客奖励中的10种重复发奖异常情况，并自动划分风险等级。

你
@数据字典.md @项目背景及审计计划.md

请参考审计计划中"4.2.1 重复发奖检测规则"的10种异常情况和"4.2.2 风险等级划分"，帮我写一个完整的检测SQL。

要求：
1. 用CTE结构，每种情况单独一个CTE
2. 最终输出每笔异常流水的明细，包含：违规类型、风险等级、审计建议
3. 按风险等级从高到低排序

AI生成的SQL结构示意（实际产出650行，以下为骨架）：

-- 10种异常情况检测 + 风险分级
WITH base AS (
  -- 筛选基础数据
  SELECT * FROM bi_inter.hk_prize_order_info_df
  WHERE ...
),
type1 AS (
  -- 情况1：单活动领奖流水 > 2笔
  SELECT user_id, act_config_sign
  FROM base GROUP BY ... HAVING COUNT(*) > 2
),
type2 AS ( -- 情况2：单活动金额重复 ... ),
-- ... type3 到 type10 ...
type10 AS ( -- 情况10：跨活动二阶奖励重复 ... ),

abnormal_users AS (
  -- 合并所有异常用户
  SELECT user_id FROM type1 UNION ... UNION SELECT user_id FROM type10
)
SELECT
  risk_level,         -- 严重异常/高风险/中风险/需人工审核
  violation_type,     -- 情况1+3:单活动>2笔且跨活动>2笔
  audit_suggestion,   -- 【紧急】金额严重异常，建议立即核查
  ...全部明细字段
ORDER BY risk_level, total_reward_amount DESC;
实际成果 这个SQL最终识别出 1,375条 异常发奖记录，涉及通用新客奖励的重复发奖问题。加上其他类型检查，HK市场共发现约 2,013条 异常，实际损失约 20万港币。

4. 增加智能标签：按"风控等级 + 发奖金额"组合判断，标注"建议复核" / "需关注" / "正常"

AI会在已有SQL基础上新增CTE来关联风控表，精准添加CASE WHEN标签列，自动处理表关联和字段映射。

这个案例说明了什么？ 即使是几百行的复杂SQL，AI也能理解现有代码上下文，并在正确的位置添加新逻辑。你只需要把业务规则说清楚——比如"风险等级4的用户如果核销金额超过2000就标注为需复核"——AI负责翻译成代码。这就是"不会写代码也能做数据分析"的核心。

使用原则与注意事项

提示词（Prompt）的黄金法则

法则说明示例
1. 给背景 告诉AI你的角色和目的 "我是内审人员，需要检查重复发奖"
2. 喂上下文 用 @ 引用数据字典、现有SQL "@数据字典.md @活动统计.sql"
3. 说具体 给出表名、字段名、筛选条件 "发奖表是 bi_inter.hk_prize_order_info_df，user_id关联员工表的uid"
4. 分步走 先提核心需求，再逐步追加 先写基础查询 → 追加标签 → 追加排序
5. 要验证 AI输出后，检查逻辑是否正确 先用小数据集测试SQL，确认结果合理

必须牢记的红线

1. 数据安全（公司有明确规定） 根据公司AI编程安全规范，以下内容禁止进入AI平台：
　• 加密密钥、认证凭证、数据库连接字符串
　• 包含风控策略数据的代码
　• 鉴权验票、密码解析逻辑
　• 虚拟币业务和银行业务代码
简单记忆：代码可以给AI看，但密钥、密码、风控数据绝对不行。
2. AI会"一本正经地胡说" AI可能生成语法正确但逻辑错误的SQL，可能编造不存在的表名和字段名，可能在报告中虚构数据。所有AI输出必须经过人工验证。
3. 专业判断不能外包 AI可以帮你分析数据、发现异常，但结论、风险判断、改善建议的最终决定权在你手中。AI是工具，不是决策者。
4. 不要盲目信任"看起来很专业"的输出 AI越是输出格式工整、措辞专业的内容，越容易让人放松警惕。始终保持专业怀疑态度。

推荐的工作流程

明确需求 — 在动手之前，先想清楚"我要查什么、为什么查、期望看到什么结果"
准备上下文 — 整理好数据字典、表结构、现有SQL等参考材料，放在项目目录里
向AI提需求 — 用自然语言描述，附上 @ 引用，先给核心需求
审核AI输出 — 检查SQL逻辑、字段名、筛选条件是否正确
小范围测试 — 加 LIMIT 100 先跑一下，看结果是否符合预期
迭代完善 — 根据测试结果，向AI提出修改要求，反复迭代直到满意
保存和归档 — 将最终SQL存入项目对应目录，结果导出CSV保存

总结：你今天学到了什么

项目阶段AI能帮你做什么你负责什么
前期 梳理工作思路、理解数据结构、组织文件结构 确认框架合理性、补充业务经验
中期 编写SQL查询、迭代优化、数据处理Python脚本 验证逻辑和结果的正确性
后期 归纳发现、起草报告段落、制作展示材料 审核事实准确性、做出专业判断

AI负责"跑腿"，你负责"动脑" — 这才是正确的人机协作方式。

除了基本的对话和代码生成，Cursor还支持两种强大的扩展机制，让AI能直接连接外部系统和学习特定技能。

MCP Model Context Protocol — 让AI直接操作外部工具

MCP（模型上下文协议）是一种标准化协议，让Cursor的AI能直接调用外部服务的API。简单来说，装了MCP就像给AI装了一个"手"，它不再只是给你建议，而是能直接帮你操作。

Step 1：打开Cursor设置（Ctrl+Shift+J 或菜单 File → Preferences → Cursor Settings）
